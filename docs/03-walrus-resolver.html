Showcase 03 · IPNS replacement

An ENS-gated mutable pointer that replaces IPNS in 35 lines.

IPNS is the single most loudly broken IPFS primitive — Pinata itself is openly pitching its EVM "IPCM" contract as a replacement. WalrusResolver takes that same idea and connects it to ENS ownership: the owner of an ENS name can park a Walrus blob pointer on-chain, anyone can resolve it in one eth_call, and a permissionless keeper keeps the underlying blob alive forever.

Sources: WalrusResolver.sol · keeper.ts Status: implemented · 8 unit tests
01 · the pain it answers

"IPNS is the most loudly broken primitive."

"Pinata is now openly pitching its smart-contract-based IPCM (IPFS CID Mapping) on EVM L2s like Base as a replacement for IPNS — a remarkable position from inside the IPFS commercial ecosystem." Kubo 0.34 release notes confirmed the scale: "Many complaints about IPNS being slow are tied to the default --ttl in ipfs name publish, which was set to 1 hour." ipfs-pain.md §6 — IPNS broken

The mental model IPNS sells — "an ENS-name-like mutable pointer to a CID" — is exactly what builders want; the part that fails is the resolution mechanism (DHT random walk, pubsub propagation, TTL tuning). WalrusResolver lifts the pointer onto EVM and the bytes onto Walrus. The pointer is one eth_call, sub-200 ms. The bytes are a content-addressed Walrus blob served by an aggregator over plain HTTPS. Both halves get observably better.

03 · read along

The contract, the authorization check, and the keeper loop

step 1 · the storage shape

WalrusResolver.sol — three fields per name, indexed event for history

The Pointer struct carries three pieces: the Walrus blobId (content address — what the aggregator URL needs), the suiObjectId (so the keeper knows which Sui object's end_epoch to extend), and a bytes8 contentType shortcode for clients that want to validate MIME before parsing.

No on-chain version array. History is reconstructable via the indexed WalrusBlobChanged event — a single SSTORE per update, no quadratic storage growth. If you need "view at block N," index the event log; if you don't, you pay nothing.

~35 LOC sol 8 unit tests ENS-gated
showcases/contracts/src/WalrusResolver.sol (excerpt) view source ↗ 
contract WalrusResolver is IERC165 {
    struct Pointer {
        bytes32 blobId;       // Walrus content address
        bytes32 suiObjectId;  // on-Sui Blob object id
        bytes8  contentType;  // e.g. "text/md", "app/json"
    }

    IENS public immutable ens;
    mapping(bytes32 node => Pointer) private _pointers;

    event WalrusBlobChanged(
        bytes32 indexed node,
        bytes32 blobId,
        bytes32 suiObjectId,
        bytes8  contentType,
        uint64  at
    );

    function setWalrusBlob(
        bytes32 node,
        bytes32 blobId,
        bytes32 suiObjectId,
        bytes8  contentType
    ) external {
        _requireAuthorized(node);
        _pointers[node] = Pointer(blobId, suiObjectId, contentType);
        emit WalrusBlobChanged(
          node, blobId, suiObjectId, contentType, uint64(block.timestamp));
    }
}

step 2 · authorization

_requireAuthorized — ENS owner OR operator-approved

One internal helper handles the whole permissioning story. The check is literally ens.owner(node) plus ens.isApprovedForAll — the same semantics ENS itself enforces in ENSIP-1. If the underlying .eth name transfers, the new owner can immediately overwrite the pointer; no separate claim step, no orphaned state.

This is the leverage ENS-gating gives you: name ownership and pointer ownership are the same fact. Squatting becomes ENS's problem, not yours.

ENSIP-1 semantics ~7 LOC
showcases/contracts/src/WalrusResolver.sol (excerpt) view source ↗ 
function _requireAuthorized(bytes32 node) internal view {
    address nodeOwner = ens.owner(node);
    require(
        nodeOwner == msg.sender ||
          ens.isApprovedForAll(nodeOwner, msg.sender),
        "WalrusResolver: not authorized"
    );
}

step 3 · the keeper

keeper.ts — read pointer, inspect Sui Blob, extend if needed

The keeper is a ~100-line Node script: for every name in KEEPER_NAMES, read the on-chain pointer, fetch the Sui Blob object to inspect its end_epoch, and call executeExtendBlobTransaction when the remaining window drops below EXTENSION_THRESHOLD. Per-name failures are caught locally so one bad name doesn't abort the cron tick.

It's permissionless on purpose: anyone willing to pay WAL can keep someone else's blog alive. The EVM pointer never moves; only the Sui-side lifetime extends.

~100 LOC @mysten/walrus 1.x cron-friendly
showcases/03-walrus-resolver/keeper/keeper.ts (excerpt) view source ↗ 
const currentEpoch = Number(
  (await sui.walrus.systemState()).committee.epoch
);

for (const name of names) {
  try {
    const node = namehash(name);
    const [, suiObjectId] = await evm.readContract({
      address: resolverAddress, abi: WALRUS_RESOLVER_ABI,
      functionName: "walrusBlob", args: [node],
    });
    if (suiObjectId === ZERO) continue;

    const blob = await sui.getObject({
      id: suiObjectId, options: { showContent: true }
    });
    const endEpoch = Number(
      blob.data?.content?.fields?.storage?.fields?.end_epoch);
    const epochsLeft = endEpoch - currentEpoch;

    if (epochsLeft >= threshold) continue;

    await sui.walrus.executeExtendBlobTransaction({
      blobObjectId: suiObjectId,
      epochs: extendBy,
      signer: keypair,
    });
  } catch (err) {
    // per-name isolation — one bad name doesn't kill the tick
    console.error(`[error] ${name}:`, err);
  }
}
04 · how it dodges IPNS's failure modes

The migration is the contract

IPNS today WalrusResolver
Resolution does a DHT random walk; first-byte often > 30 s. Pointer is one eth_call (< 200 ms).
Pubsub-based update propagation is unreliable. Updates land in one EVM transaction with on-chain provenance.
Pinning expires silently when the pinning service forgets. The Sui Blob object id is on-chain — anyone can keep it alive via the keeper.
Squatting on names is trivial; no canonical owner. Names are ENS-gated; sale of the ENS name carries the pointer with it.
No useful audit trail of past pointer values. Indexed WalrusBlobChanged events — full history reconstructable from logs.

06 · what to try next

Exercises

Compute namehash("blog.vitalik.eth") by hand

The contract takes bytes32 node, not a string. Use viem's namehash() (the keeper does) and compare against an EIP-137 reference implementation to internalize how subdomains roll up.

Stand up a fake ENS for testing without mainnet

MockENS in the test file is the smallest viable ENS surface: owner() + isApprovedForAll(). Deploy it to Anvil alongside WalrusResolver and you can exercise the full set/read flow locally.

Set up an operator approval and have the keeper drive everything

Call ens.setApprovalForAll(operator, true) and let the operator address handle setWalrusBlob for you. Pattern matches how teams already give CI bots write access to ENS records.

Read the contentType bytes8 carefully

It's intentionally not a full MIME. Showcase 05's asciiTrim + startsWith("app/json") check is the convention. Pick a shortcode grammar your readers can rely on before launching.

Reconstruct full history from the event log

Subscribe to WalrusBlobChanged(indexed node, ...) with viem's getLogs and you have a full mutable-pointer history without ever adding a storage array. That's the trade the contract is making.
