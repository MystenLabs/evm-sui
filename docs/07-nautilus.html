Showcase 07 · implemented · the path past EVM

When an EVM contract can't verify the off-chain part, leave EVM.

Showcases 01–06 swap the storage layer behind a Solidity contract. This one swaps the computation layer behind it. A Rust enclave fetches a CoinGecko price, BCS-encodes it, signs it with an ephemeral ed25519 key; a 60-line Sui Move package verifies the signature. The full loop runs on a laptop — no AWS, no Marlin Oyster, free.

Source: showcases/07-nautilus/ Stack: Rust · axum · ed25519-dalek · Sui Move 2024 · BCS Upstream: MystenLabs/nautilus
01 · the pain

The leading EVM solution for this is in BETA — and the vendor says so on every page.

vendor disclaimer
notes/NAUTILUS_EVM.md · §3 — Secret management
"Chainlink Functions is still in BETA. The use of secrets in your requests is an experimental feature that may not operate as expected and is subject to change. Use of this feature is at your own risk and may result in unexpected errors, possible revealing of the secret as new versions are released, or other issues."
— Chainlink Functions docs, every secrets-related page

That's not the only one. samczsun published the canonical price-oracle essay in 2020 and the same mistakes kept shipping: $403.2M lost across 41 oracle attacks in 2022 alone, $117M of it Mango Markets. Ronin lost $624M because five-of-nine validator keys lived on infrastructure one operator controlled. The April 2026 LayerZero/KelpDAO exploit drained $293M because a 1-of-1 DVN read from internal RPCs an attacker had poisoned. The pattern is the same: a smart contract trusts an off-chain value or signer set, the off-chain side has no cryptographic attestation, the attacker rewrites the value or the signers. The investigation under notes/NAUTILUS_EVM.md walks twelve such categories with direct dev voices.

02 · the shape

One Rust binary signs. One Move package verifies. BCS in the middle.

The enclave (Rust + axum) exposes POST /process_data. It fetches a price from CoinGecko, builds a PriceResponse, wraps it in an IntentMessage<PriceResponse> (intent byte + timestamp + payload), BCS-encodes the whole thing, signs with its ephemeral ed25519 key, and returns { response, signature }.

The Move package mirrors the payload struct field-for-field. update_price rebuilds the same IntentMessage, BCS-encodes it, and calls ed25519_verify against the enclave's registered pubkey. The pubkey was pinned upfront when register_enclave consumed a Nitro attestation document whose PCRs match the registered EnclaveConfig.

Locally the attestation half is bypassed (all-zero PCRs + deterministic key). The BCS + signature half stays real, so the on-chain verifier is the same one that ships to production. 

// move/price-oracle/sources/oracle.move
public fun update_price<T>(
    coin_id: String,
    vs: String,
    price_micro: u64,
    timestamp_ms: u64,
    sig: &vector<u8>,
    enclave: &Enclave<T>,
    ctx: &mut TxContext,
): PriceFeed {
    let payload = PricePayload { coin_id, vs, price_micro };
    let ok = enclave.verify_signature(
        PRICE_INTENT, timestamp_ms, payload, sig
    );
    assert!(ok, EInvalidSignature);
    PriceFeed { id: object::new(ctx), coin_id, vs, price_micro, timestamp_ms }
}

03 · try it locally

From `cargo run` to a PriceFeed on testnet in five steps.

01

Compile and run the enclave

cd showcases/07-nautilus/enclave
cargo run
# → ENCLAVE_PUBKEY_HEX=4cb19c0bb37b1f1b95a3...   ← save this
# → listening on http://0.0.0.0:3000

Default features include dev-key, which replaces the per-boot random ed25519 key with a fixed seed ([42u8; 32]). The printed pubkey is stable across restarts so you can register it once and reuse forever. The nitro feature is off — no AWS Nitro driver dep, compiles clean on macOS.

02

Confirm signing works

curl -fsS -H 'Content-Type: application/json' \
     -d '{"payload":{"coin_id":"sui","vs":"usd"}}' \
     http://localhost:3000/process_data | jq .
{
  "response": {
    "intent": 0,
    "timestamp_ms": 1747948923174,
    "data": { "coin_id": "sui", "vs": "usd", "price_micro": 3450000 }
  },
  "signature": "b75d2d44...0a"
}

/get_attestation intentionally returns 400 locally — it needs the NSM driver, which exists only inside a real Nitro Enclave.

03

Publish both Move packages

sui client switch --env testnet
sui client faucet

cd ../move/enclave   && sui move build && sui client publish
# → record ENCLAVE_PACKAGE_ID

cd ../price-oracle  && sui move build && sui client publish
# → record APP_PACKAGE_ID, CAP_ID, ENCLAVE_CONFIG_ID

The vendored enclave package is the generic Nautilus verifier — same source as upstream, do not modify. The price-oracle package is the app surface; its init creates an EnclaveConfig with all-zero PCRs for local dev.

05

Close the loop

export APP_PACKAGE_ID=0x...
export ENCLAVE_OBJECT_ID=0x...

cd showcases/07-nautilus
./scripts/post-price.sh sui usd

The script curls /process_data, extracts the signature + timestamp + price, and submits to update_price. If Rust and Move agree on BCS field order, and the pubkey on-chain matches the one signing, a PriceFeed shared object lands on testnet. View it on suivision.xyz.

04 · verify

What "good" looks like.

  • cargo test --lib passes — the pinned BCS hex 0x000068e5cf8b010000037375690375736490a4340000000000 matches.
  • sui move test in move/price-oracle/ passes — Move emits the same hex.
  • If you reorder a field in either PriceResponse or PricePayload, both tests fail loud. That's the entire point.
  • The signed response's response.data matches what curling CoinGecko directly returns (modulo the *1_000_000 integer scaling).
  • The on-chain PriceFeed object's fields match the values the script passed in.
05 · what this is NOT

Limits to set expectations.

  • Not real attestation. All-zero PCRs prove nothing about which binary signed. Production needs a Nitro instance, a reproducible build, and the standard register_enclave.sh flow that consumes the NSM document.
  • Not production-safe. The deterministic seed [42u8; 32] is a development convenience — anyone who reads the source can recover the signing key. Strip --features=dev-key before shipping.
  • Not an EVM showcase. Verifying a Nitro attestation in Solidity costs ~70M gas after two rounds of optimization (Marlin's numbers). The Sui-Move ed25519 primitive makes per-payload verify trivial; that asymmetry is why this showcase lives on Sui. EVM developers wanting the same shape should look at Marlin Oyster's EVM-side verifier.
  • Not a complete framework port. This vendors only what's needed for one app. The full Nautilus repo includes Twitter, Weather, and Seal-policy examples; the reproducible-build pipeline; the host-init protocol for sealed secrets; the official scripts. Use MystenLabs/nautilus upstream when you've outgrown this scaffold.
06 · honest contrast

Where Nautilus wins — and where it doesn't.

Wins outright

  • Greenfield Sui apps that need verifiable off-chain compute. First-party Mysten tooling, tightest BCS↔Move handshake of any TEE platform.
  • App-specific oracles where the fetch logic is yours (social, sports, niche feeds). Cleaner than Chainlink Functions' BETA secrets path.
  • Confidential matching engines on Sui — public ledger, private matching, settlement enforced by Move.
  • AI agents whose code identity matters: PCRs commit to which binary holds the agent's wallet key.
  • Reproducible-build audit story. The PCR is a function of the source.

Wrong tool for the job

  • You must remain on Ethereum or an EVM L2 for liquidity. Pick Phala, Marlin Oyster (EVM-side), Oasis Sapphire, or — accepting the BETA caveat — Chainlink Functions.
  • Fully confidential EVM by default. Oasis Sapphire is purpose-built; Solidity ports cleanly.
  • MEV infrastructure for Ethereum. Flashbots SUAVE / BuilderNet owns this lane.
  • Multi-vendor TEE diversity (AWS + GCP + Intel + AMD). Nautilus is AWS Nitro–first; Phala's roadmap is broader.
  • Pure VRF or pure timestamping. Sui has native primitives; Nautilus is overkill.
