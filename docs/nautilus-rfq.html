Walrus - EVM Integrations nautilus / N1 · Sealed RFQ
nautilus · N1

Sealed RFQ: the best price wins, nobody sees it first.

A Nautilus enclave running inside AWS Nitro collects maker quotes over TLS, picks the lowest price for the taker, signs the result with Ed25519, and posts the proof to Sui. No one — not the operator, not other makers — can see the quotes before the match. A Solidity stub shows where EVM settlement would plug in.

Stack: Rust · Move · Solidity stub Source: nautilus/ Local: Sui localnet · Anvil · enclave on localhost:3000
01 · the problem

Maker quotes are public, so bots front-run every fill.

In a traditional RFQ system the taker broadcasts a request, makers respond with prices, and the best quote wins. But on any transparent mempool — EVM or otherwise — the quotes are visible before execution. MEV bots can copy the winning price minus one tick, sandwiching the maker or front-running the taker.

The fix is obvious: run the matching off-chain in a trusted execution environment so quotes are never visible to anyone except the enclave code. The result is a signed attestation that this price was the best among N quotes received before the deadline.

MEV in plain-text RFQ. If quotes travel through a public mempool (or even a relay with read access), any observer can extract the winning price before settlement. Sealed matching inside a TEE makes the quotes invisible until the match is finalized.
02 · the shape

User, Enclave, Sui, then (optionally) EVM.

Taker + Makers
TLS
Nautilus Enclave
signs
Sui Move (verify)

Three zones: (1) users submit quotes to the enclave over TLS — the enclave is the only party that sees them. (2) The enclave picks the best price, BCS-encodes the result, and signs it with its ephemeral Ed25519 key. (3) The signed payload is submitted to the Sui Move contract, which calls verify_signature() against the enclave's registered public key and PCR measurements. A Walrus blob stores the full attestation bundle for independent audit.

The Solidity stub (RfqSettlement.sol) shows a fourth optional zone: the same result posted to EVM via a msg.sender == enclaveSigner gate. Production would use EIP-712 ecrecover.

03 · move contract

RfqPayload and settle_rfq.

The rfq::rfq module defines a RfqPayload struct whose BCS field order must match the Rust-side struct exactly (BCS is positional). On settlement the contract reconstructs the payload, calls enclave.verify_signature(), and if valid creates a shared SettledRfq object anyone can read.

The init function creates an EnclaveConfig with placeholder PCRs (all zeroes for local dev) and transfers the admin Cap to the deployer. 

// move/rfq/sources/rfq.move

    price: u64,
    walrus_proof_blob_id: vector<u8>,

04 · enclave logic

Argmin: lowest price from non-expired quotes.

The Rust enclave server exposes a single POST /process_data endpoint. It receives an RfqRequest containing the taker's order and N maker Quote structs (each with a price and expiry). The matching function filters expired quotes, then returns the one with the lowest price — a simple min_by_key.

The result is BCS-serialized into an IntentMessage<RfqPayload> and signed with the enclave's ephemeral Ed25519 key. The response includes both the signed payload and the hex signature. 

// enclave-server/src/rfq/mod.rs
fn select_best_quote(
    quotes: &[Quote], now_ms: u64
) -> Option<&Quote> {
    quotes
        .iter()
        .filter(|q| q.expiry_ms > now_ms)
        .min_by_key(|q| q.price)
}

// Quote struct
pub struct Quote {
    pub maker: Vec<u8>,
    pub price: u64,
    pub expiry_ms: u64,
}
05 · solidity stub

RfqSettlement: the EVM side.

// contracts/src/RfqSettlement.sol — showcase stub
contract RfqSettlement {
    struct Settlement {
        bytes32 rfqId;
        address taker;
        address winningMaker;
        address baseToken;
        address quoteToken;
        uint256 amount;
        uint256 price;
        bytes32 proofBlobId;   // Walrus attestation bundle
        uint64  settledAt;
    }

    address public immutable enclaveSigner;

    function settle(bytes32 rfqId, address taker, ...)
        external;

    event Settled(bytes32 indexed rfqId, address taker,
                 address winningMaker, uint256 price);
}

The stub uses msg.sender == enclaveSigner as a simplified auth check. Production would verify an EIP-712 typed-data signature via ecrecover so the enclave does not need to hold ETH for gas.

06 · run locally

Build, start, verify — five commands.

01

Build the Move packages

cd nautilus/move/enclave && sui move build
cd ../rfq && sui move build
02

Build and test the Rust enclave server

cd nautilus/enclave-server
cargo test --features rfq
cargo run --features rfq
# → Enclave pubkey: ab12...
# → listening on 0.0.0.0:3000

Set DETERMINISTIC_KEY=<64-char-hex> for a stable pubkey across restarts (dev only).

03

Build and test the Solidity stubs

cd nautilus/contracts
forge install --no-git foundry-rs/forge-std
forge test -vv
04

Submit a test RFQ to the enclave

curl -X POST http://localhost:3000/process_data \
  -H "Content-Type: application/json" \
  -d '{"payload":{"rfq_id":[1],"base_token":[16],"quote_token":[32],"amount":1000,"quotes":[{"maker":[1],"price":500,"expiry_ms":9999999999999},{"maker":[2],"price":300,"expiry_ms":9999999999999}]}}'
# → { "response": { "intent": 0, "timestamp_ms": ..., "data": { "price": 300, ... } }, "signature": "..." }
07 · verify

What "good" looks like.

  • cargo test --features rfq passes all RFQ tests including BCS round-trip and best-quote selection.
  • The /process_data endpoint returns the quote with the lowest price among non-expired entries.
  • forge test -vv passes the RfqSettlement tests — settle once succeeds, duplicate reverts.
  • The BCS hex from the Rust test matches what the Move RfqPayload would produce for the same inputs.
