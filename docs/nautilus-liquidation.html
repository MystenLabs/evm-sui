nautilus · N2

Sealed liquidation: the highest bonus wins, bots can't peek.

When a DeFi position goes under-collateralised, liquidation bots race to claim it. The fastest bot wins, not the one offering the best terms. A Nautilus enclave collects sealed bids over TLS, picks the bidder offering the highest liquidation bonus (best for the protocol), signs the result, and posts proof to Sui. Walrus stores the full attestation bundle.

Stack: Rust · Move · Solidity stub Source: nautilus/ Local: Sui localnet · Anvil · enclave on localhost:3000
01 · the problem

Liquidation is a speed contest, not a fairness contest.

On Aave, Compound, and every EVM lending protocol, liquidation is first-come-first-served. Bots monitor the mempool, detect under-collateralised positions, and submit liquidation transactions with high gas to win the race. The protocol gets whatever bonus the fastest bot chose — often the minimum.

A sealed-bid liquidation auction inside a TEE flips the incentive: bidders compete on bonus offered to the protocol, not on latency. The enclave picks the highest liquidation_bonus_bps among non-expired bids, signs the result, and both Sui and EVM can verify it.

MEV in liquidation. Priority gas auctions mean the protocol always gets the minimum viable liquidation bonus. Sealed bids inside a TEE force bidders to compete on value, not speed.

The architecture mirrors the RFQ example: bidders submit sealed bids to the enclave over TLS. The enclave filters expired bids, selects the one with the highest liquidation_bonus_bps (argmax instead of argmin), BCS-encodes the result into a LiquidationPayload, and signs it. The Move contract verifies the signature and creates a shared SettledLiquidation object. The Solidity stub records the same outcome on EVM.

03 · move contract

LiquidationPayload and execute_liquidation.

The liquidation::liquidation module defines a LiquidationPayload carrying the position details and the winning bid. The execute_liquidation function reconstructs the payload, verifies the enclave signature, and shares a SettledLiquidation object on-chain.

Key fields: position_id (the under-collateralised position), winning_bidder, collateral_amount / debt_amount, and liquidation_bonus_bps (basis points the winner offered to the protocol). 

// move/liquidation/sources/liquidation.move
public struct LiquidationPayload has copy, drop {
    position_id: vector<u8>,
    borrower: vector<u8>,
    winning_bidder: vector<u8>,
    collateral_token: vector<u8>,
    debt_token: vector<u8>,
    collateral_amount: u64,
    debt_amount: u64,
    liquidation_bonus_bps: u64,
    walrus_proof_blob_id: vector<u8>,
}

04 · enclave logic

Argmax: highest bonus from non-expired bids.

The Rust enclave receives a LiquidationRequest containing the position details and N Bid structs (each with a liquidation_bonus_bps and expiry_ms). The matching function filters expired bids, then returns the one with the highest bonus — a simple max_by_key. The protocol benefits because it extracts maximum value from the liquidation event.

The result is wrapped in an IntentMessage<LiquidationPayload>, BCS-serialized, and Ed25519-signed. 

// enclave-server/src/liquidation/mod.rs
fn select_best_bid(
    bids: &[Bid], now_ms: u64
) -> Option<&Bid> {
    bids.iter()
        .filter(|b| b.expiry_ms > now_ms)
        .max_by_key(|b| b.liquidation_bonus_bps)
}

// Bid struct
pub struct Bid {
    pub bidder: Vec<u8>,
    pub liquidation_bonus_bps: u64,
    pub expiry_ms: u64,
}
05 · solidity stub

LiquidationSettlement: the EVM side.

// contracts/src/LiquidationSettlement.sol — showcase stub
contract LiquidationSettlement {
    struct Liquidation {
        bytes32 positionId;
        address borrower;
        address liquidator;
        address collateralToken;
        address debtToken;
        uint256 collateralSeized;
        uint256 debtRepaid;
        uint16  bonusBps;        // winning bonus
        bytes32 proofBlobId;     // Walrus attestation
        uint64  settledAt;
    }

    function liquidate(
        bytes32 positionId, address borrower, ...
    ) external;

    event Liquidated(bytes32 indexed positionId,
        address borrower, address liquidator,
        uint256 collateralSeized, uint256 debtRepaid);
}

Same auth simplification as the RFQ stub: msg.sender == enclaveSigner. The proofBlobId links to the Walrus blob containing the enclave measurement, position snapshot, oracle prices, and the full signed computation trace.

06 · run locally

Build, start, verify.

01

Build the Move packages

cd nautilus/move/enclave && sui move build
cd ../liquidation && sui move build
02

Build and test the Rust enclave server

cd nautilus/enclave-server
cargo test --no-default-features --features liquidation
cargo run --no-default-features --features liquidation
# → Enclave pubkey: cd34...
# → listening on 0.0.0.0:3000

Only one feature can be active at a time. Use --no-default-features to disable the default rfq feature.

03

Build and test the Solidity stubs

cd nautilus/contracts
forge test -vv --match-contract LiquidationSettlement
04

Submit a test liquidation bid

curl -X POST http://localhost:3000/process_data \
  -H "Content-Type: application/json" \
  -d '{"payload":{"position_id":[221],"borrower":[1],"collateral_token":[16],"debt_token":[32],"collateral_amount":5000,"debt_amount":4000,"bids":[{"bidder":[1],"liquidation_bonus_bps":200,"expiry_ms":9999999999999},{"bidder":[2],"liquidation_bonus_bps":500,"expiry_ms":9999999999999}]}}'
# → { "response": { ..., "data": { "liquidation_bonus_bps": 500, "winning_bidder": [2], ... } }, "signature": "..." }
07 · verify

  • cargo test --no-default-features --features liquidation passes all tests including BCS round-trip and best-bid selection.
  • The /process_data endpoint returns the bid with the highest liquidation_bonus_bps among non-expired entries.
  • forge test -vv passes the LiquidationSettlement tests — liquidate once succeeds, duplicate reverts.
