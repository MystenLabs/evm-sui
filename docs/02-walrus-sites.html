Showcase 02 · frontend hosting

Ship the dApp itself on Walrus, not IPNS.

The bytes of an EVM dApp's frontend are exactly the kind of asset that broke when Cloudflare sunsetted its public IPFS gateway in 2024 and Uniswap had to rewrite every reference to it. Walrus Sites publishes a static build with one command, parks it as a Sui object, and lets the public wal.app portal resolve it for any visitor — without IPNS, DNSLink, or a single hosted-gateway dependency.

Sources: publish.sh Status: scaffolded Walrus docs: docs.wal.app/sites
01 · the pain it answers

ENS contenthash works; the IPFS network underneath doesn't.

"Cloudflare's public IPFS gateway (cf-ipfs.com) was fully decommissioned on August 14, 2024." Uniswap GitHub issue interface#7753: "All references to Cloudflare's IPFS gateway should be updated because all these links will be broken soon." And Pinata's VP of Product in January 2025: "IPNS has been known to be slow and hard to work with. One of the biggest problems with IPNS is the speed." ipfs-pain.md §2, §6 — gateways and IPNS

The state of the art for "decentralized dApp frontend" in 2026 is a paid Pinata gateway + ENS contenthash, with an IPNS record that nobody trusts and a DNSLink TXT record that propagates whenever DNS feels like it. That works until any one of those services blinks. Walrus Sites replaces the IPFS-network half with one Sui transaction and a portal that doesn't DHT-walk.

03 · read along

The whole showcase is one bash script

publish.sh is the entire surface area — it just wraps site-builder publish with the conventions you'll want (sane defaults, input validation, and a printed next-steps block for the SuiNS + ENS hookup).

step 1 · the publish command

publish.sh — validate inputs, call site-builder, print the SuiNS recipe

The interesting work is the site-builder call. Everything around it (regex check on SITE_NAME, network whitelist, epochs sanity check) exists because the script is the kind of thing teams paste into a CI runbook — fail loudly before publishing, not after.

The output includes a site_object_id on Sui. That's the only artifact you need to link your SuiNS name; the contents are content-addressed and verifiable by anyone with the id.

#!/usr/bin/env bash
# Publish a Walrus Site for the directory passed in $1.
set -euo pipefail

SITE_DIR=${1:-./example-site}
EPOCHS=${EPOCHS:-200}
SITE_NAME=${SITE_NAME:-"showcase-02-demo"}
NETWORK=${NETWORK:-testnet}

# reject names that would surprise the CLI parser or look unsafe in logs
if ! printf '%s' "$SITE_NAME" | grep -Eq '^[A-Za-z0-9._-]{1,64}$'; then
  echo "SITE_NAME must match [A-Za-z0-9._-]{1,64}" >&2; exit 1
fi

site-builder \
  --context "$NETWORK" \
  publish "$SITE_DIR" \
  --epochs "$EPOCHS" \
  --site-name "$SITE_NAME"

# ↑ site-builder prints something like:
#     site_object_id: 0xa1b2c3...
# Link that id to your SuiNS name in https://suins.io
# Visit https://<your-name>.wal.app to confirm.

step 2 · what you publish

example-site/index.html — a placeholder, but the real one is your Next.js build

The repo ships a one-file placeholder so the script has something to publish without shipping a full build artifact in source control. For a real dApp, site-builder publish ./out (Next.js export) or ./dist (Vite) gives you the whole SPA on Walrus.

What's not here is any IPFS-era plumbing — no pinning service token, no IPNS publish step, no DNSLink propagation wait. Just the static build and a single CLI call.

04 · run it locally

Install site-builder, publish, then point a name at it

Publish

terminal
# install the Walrus site-builder CLI
# https://docs.wal.app/sites/getting-started/installing-the-site-builder/

# ensure a testnet Sui keypair with gas SUI + WAL is configured
cat ~/.config/walrus/sites-config.yaml

# publish the placeholder (or your own build dir)
cd showcases/02-walrus-sites
./publish.sh ./example-site

# → site_object_id: 0x...
# → resolves at <base36-of-id>.wal.app immediately

Hook it up

SuiNS + ENS
# 1. Link a SuiNS name → site object id (one Sui tx).
#    https://suins.io → your name → "Set walrus site"
#    Now https://your-name.wal.app resolves the site.

# 2. (optional) Bridge from ENS via DNSLink.
#    At your DNS host, on _dnslink.yourdapp.eth set a TXT:
#      dnslink=/https/your-name.wal.app
#    Then set ENS contenthash to DNSLink('yourdapp.eth').

# 3. (optional alt) CNAME path:
#    A regular DNS CNAME from app.yourdomain.com to
#    your-name.wal.app, then point ENS at app.yourdomain.com.
05 · what to try next

Exercises

Publish a real Next.js build, not the placeholder

cd web && pnpm build && pnpm export (or your framework's equivalent) and pass the output dir to publish.sh. Confirm wallet connect and contract reads still work when the dApp is served from your-name.wal.app.

Measure first-byte latency vs an IPFS gateway

curl -w '%{time_starttransfer}\n' -o /dev/null -s against your wal.app URL and against any public IPFS gateway hosting equivalent content. The Walrus number should be stable across reloads; the IPFS one famously isn't (see ipfs-pain.md §7).

Push an update and re-publish

Change example-site/index.html and re-run the script. Note the new site_object_id — that's how Walrus Sites handle "mutability": content-addressed bytes plus a SuiNS pointer you update in one tx. No IPNS, no propagation window.

Read the Walrus Sites docs end-to-end

docs.wal.app/sites covers things this showcase doesn't: site-builder config schema, redirect/headers files, and per-network portals. Useful once you're past the placeholder.
