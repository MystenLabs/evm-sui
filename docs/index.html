Builder report · 2026-05-14

Six precise dApps Walrus can ship to convert IPFS pain into adoption.

Web3 builders in 2026 don't dislike content addressing — they dislike the IPFS network. CIDs survive, ENS contenthash is non-negotiable, and verifiability is non-optional. What's collapsing is the operational promise: free public goods that deprecated, pinning vendors that churn SDKs, and IPNS that quietly broke everyone's mutable pointers. This site names six low-hanging fruits where Walrus can answer a specific, already-voiced grievance with a small, opinionated example in the shape of EvmWal NFT.

02 · thesis

Content addressing detached from the network

The crucial observation in the 2026 survey is that the conversation has detached "content addressing" from "IPFS the network." Builders still want CIDs — for verifiability, ENS contenthash, on-chain references — but no longer want to participate in or trust the public P2P substrate underneath. That detachment is exactly the surface Walrus is built for.

The EvmWal NFT example demonstrates this in 200 lines: a vanilla OpenZeppelin ERC-721 storing a plain tokenURI string — identical in shape to an IPFS-backed NFT — pointing at a Walrus aggregator URL. The Solidity side is unchanged. The migration cost is one URL prefix.

Every fruit on this tree shares the same shape: keep the contract that builders already know, swap the storage layer behind a single fetch.

Verifiable retrieval, native

Walrus aggregators serve content-addressed blobs over plain HTTPS — verifiability without P2P plumbing in the browser.

On-Sui Blob objects

Each blob is a Sui object an operator can extend, delete, or transfer — replacing fragile pinning-service state with on-chain truth.

Two write paths, one read path

Public publisher PUT for zero-infra prototypes; SDK-backed operator path for production. Reads are the same anonymous aggregator GET either way.

Deterministic node assignment

Walrus's Red Stuff erasure coding fixes the "random walk" provider-discovery problem Shipyard itself cites as IPFS's structural retrieval flaw.

03 · template

The shape of a Walrus showcase

The EvmWal NFT example, already shipped in this repo, is the canonical template. Each of the six fruits below is a variation on it: one tiny Solidity contract, one or two fetch calls, one URL prefix.

Browser write — one fetch, either mode

Default mode posts to a local Next.js route that signs Sui transactions with an env-keyed operator wallet. Publisher mode PUTs directly to the public publisher and the EVM mint still works.

// from web/lib/walrus-upload.ts — fallback (publisher) path
const res = await fetch(`${WALRUS_PUBLISHER}/v1/blobs?epochs=5`, {
  method: 'PUT',
  headers: { 'Content-Type': mime },
  body: bytes,
});
const { blobId } = await res.json();

// from web/app/api/walrus/upload/handler.ts — operator (backend) path
const { blobId, blobObject } = await deps.walrusClient.walrus.writeBlob({
  blob: bytes,
  deletable: false,
  epochs: 5,
  signer: deps.keypair,
});

EVM contract — unchanged from IPFS

OpenZeppelin ERC-721 + ERC721URIStorage. The contract has no Walrus knowledge — it just holds a string. The migration from IPFS is changing what your frontend uploads to and which URL prefix it reads from.

// from contracts/src/EvmWalNFT.sol — same shape an IPFS NFT uses
function mint(string memory tokenURI_) external returns (uint256 tokenId) {
    tokenId = _nextId();
    _safeMint(msg.sender, tokenId);
    _setTokenURI(tokenId, tokenURI_);
    emit Minted(tokenId, msg.sender, tokenURI_);
}

// tokenURI_ ends up as:
// https://aggregator.walrus-testnet.walrus.space/v1/blobs/<blobId>

Walrus Sites for dApp frontend hosting

implemented · 02-walrus-sites/

The pain

"ENS contenthash is non-negotiable; pay Pinata or Filebase rather than trusting any free gateway; treat IPNS as broken." — and Uniswap interface #7753 had to rewrite "all references to Cloudflare's IPFS gateway." ipfs-pain.md §10 — the 2026 newcomer vibe

The Walrus fix

Publish a Next.js or SPA build with site-builder publish; a Sui object holds the site, a SuiNS name points at it, and the public wal.app portal resolves it for any visitor. Updates land via one Sui transaction — no IPNS, no DNSLink. The ENS bridge is a single CNAME or DNSLink record from your .eth name to the portal host.

03

Mutable pointer — the IPNS replacement

implemented · WalrusResolver.sol + keeper

The pain

"Pinata is now openly pitching its smart-contract-based IPCM (IPFS CID Mapping) as a replacement for IPNS — a remarkable position from inside the IPFS commercial ecosystem." ipfs-pain.md §6 — IPNS broken

The Walrus fix

A 35-line Solidity registry: ENS name → current blobId + Sui object id, ENS-gated update, full event log for verifiable history. Replaces IPNS with one EVM read and one aggregator GET. A small TS keeper extends the underlying Sui Blob object whenever its epoch retention drops.

trivial ~50 LOC total migration: replace ipns:// with one contract read
Shape · Solidity registry
// WalrusResolver.sol — IPNS-shaped, on-chain, no random walk
contract WalrusResolver {
  function setWalrusBlob(
    bytes32 node, bytes32 blobId, bytes32 suiObjectId, bytes8 ct
  ) external;

  function walrusBlob(bytes32 node) external view returns (
    bytes32 blobId, bytes32 suiObjectId, bytes8 contentType
  );

  event WalrusBlobChanged(
    bytes32 indexed node, bytes32 blobId, bytes32 suiObjectId, bytes8 contentType, uint64 at
  );
}

04

DAO governance proposals — Snapshot pattern

implemented · 04-dao-proposals/ + Governance.sol

The pain

"4Everland is used by Snapshot governance for Aave and Uniswap DAO votes" — bundled into the same hosted-IPFS tier that absorbs the operational pain because public gateways can't be trusted. ipfs-pain.md §9 — hosted IPFS providers

The Walrus fix

One contract storing proposalId → (proposer, blobId, deadline, yes, no). Proposal body uploaded to Walrus in publisher mode (so the proposer pays, not the DAO operator); contract stores the 32-byte blobId. Anyone reads the proposal anonymously via the aggregator. No pinning-vendor dependency, no rate-limit surprise on vote day.

straightforward ~100 LOC showcase-only voting surface migration: store blobId, not CID
Shape · proposer-pays write + DAO-readable bytes
// Governance.sol — proposal body lives on Walrus, weight on-chain
struct Proposal {
  address proposer;
  bytes32 blobId;       // Walrus blob holding the proposal markdown
  uint64  deadline;
  uint128 yes;
  uint128 no;
}
function propose(bytes32 blobId, uint64 deadline) external returns (uint256 id);
function vote(uint256 id, bool support) external;

// readers fetch the body from: aggregator/v1/blobs/<blobId>

05

Verifiable token list / dApp manifest

implemented · 05-verifiable-manifest/

The pain

"ipfs.io 85–90% of the time, it resolves … sometimes takes 5 min" — and Uniswap's interface release notes broke because public gateways flake. Token lists and dApp manifests are exactly the small, high-read, cache-friendly artifact that public-gateway flakiness ruins. ipfs-pain.md §2 — gateways

The Walrus fix

Combine showcase 03's mutable pointer with a strict JSON schema. Frontend resolves the pointer in one eth_call and fetches the manifest from a Walrus aggregator by its content-addressed blob id. The aggregator is the trust boundary — same model as fetching by CID from an IPFS gateway — but the *pointer* lookup is now deterministic instead of an IPNS lottery, which is the actual leverage over the Uniswap token-list pattern. Add Reed-Solomon-backed client-side hash verification on top with @mysten/walrus if your threat model needs it.

straightforward ~80 LOC migration: swap ipfs:// for blobId lookup
Shape · one fetch + one verification
// resolve "tokens.uniswap.eth" → current blobId → manifest JSON
const { manifest, blobId, contentType } = await resolveTokenList("tokens.uniswap.eth", {
  rpcUrl: EVM_RPC_URL,
  resolverAddress: RESOLVER_ADDRESS,
  aggregator: "https://aggregator.walrus-testnet.walrus.space",
});

// manifest is a typed Uniswap TokenList; blobId is the 32-byte source of truth.

06

NFT collection drop — one Walrus Quilt, 10 000 tokens

implemented · 06-quilted-collection/ + QuiltedCollection.sol

The pain

"OpenSea actually improved their NFT metadata reliability by 99.2% when they switched to Pinata's infrastructure" — i.e. the centralization tax for collection drops is enormous because each token needs its own pin. ipfs-pain.md §8 — decentralization theater

The Walrus fix

Pack every image + metadata file from the drop into two Walrus Quilts (images then metadata, so the metadata can reference the images' quiltId without a circular dependency). The contract stores one quiltId; tokenURI(id) deterministically returns the aggregator URL for that token's metadata. One upload per layer, 10 000 retrievable blobs, no pin-management surface.

solid ~120 LOC requires walrus CLI migration: two-quilt packer + 1-line tokenURI
Shape · quilt + deterministic tokenURI
# pack images first, then metadata referencing them
pnpm pack:quilt ./images    # → QID_IMAGES
# generate meta/N.json with image=<agg>/by-quilt-id/<QID_IMAGES>/N.png
pnpm pack:quilt ./meta      # → QID_METADATA

// QuiltedCollection.tokenURI
function tokenURI(uint256 id) public view returns (string memory) {
    return string.concat(aggregator, "/v1/blobs/by-quilt-id/", quiltId, "/", id.toString(), ".json");
}

06 · sequencing

What to ship, in what order

Ordered by leverage-per-week-of-engineering. The first three are stand-alone repos in the shape of MystenLabs/evm-nft-wal. The last three reuse infrastructure from earlier showcases.

Mutable pointer demo (showcase 03)

Smallest possible repo. Directly answers the IPNS grievance — the single most-cited structural complaint in the report. Becomes a dependency for showcases 05 and parts of 04.

≈ 1 dev-week

Walrus Sites + SuiNS frontend how-to (showcase 02)

Mostly a focused tutorial repo: a tiny Next.js app, one publish script, a SuiNS-link walkthrough, and an optional CNAME from a .eth name to wal.app. Closes the Cloudflare-gateway-sunset wound with a path Uniswap-class projects can follow.

≈ 1 dev-week

Verifiable token list (showcase 05)

Fork the mutable-pointer repo and add the manifest schema + a viewer. Speaks directly to the wallet / DEX teams already maintaining IPFS-pinned token lists.

≈ 1 dev-week

DAO proposals (showcase 04)

Larger surface — a proposer UI plus a vote-tally view. Best landed as a Snapshot adapter rather than a full Snapshot replacement; reuse the EvmWal NFT publisher path so proposers pay their own WAL.

≈ 2 dev-weeks

Quilted collection drop (showcase 06)

Pairs naturally with an NFT studio partner — Magic Eden, Sound.xyz, or a launchpad. Requires polish on the bulk packer and the patch-id URL helper, but every other piece is reused from showcase 01.

≈ 2 dev-weeks

Composite: dApp frontend reference stack

Once 02–05 exist, bundle them as a "Walrus reference dApp": Walrus Sites for hosting, mutable pointer for manifests, NFT example for tokens, DAO module for governance. This is the canonical "what does a Walrus-native dApp look like" answer.

≈ 1 sprint of integration
Methodology

Each showcase is anchored to at least one verbatim builder quote in ipfs-pain.md (the 2026 builder-sentiment survey checked into the repo). Code sketches assume the same stack as MystenLabs/evm-nft-wal: OpenZeppelin ERC-721 + ERC721URIStorage, Next.js 16 App Router, viem + wagmi, and either @mysten/walrus in a local Next.js route or the public Walrus publisher. LOC estimates are total lines for a self-contained demo repo, not just net-new code.

The report is intentionally narrow: every fruit listed is one a small DevRel team can ship in 1–2 weeks and which a mid-sized EVM dApp can adopt in days. Larger Walrus-native designs (data marketplaces, Walrus Relays, Nautilus-attested uploads) are out of scope on purpose — they're the next tree, not the low-hanging fruit.
