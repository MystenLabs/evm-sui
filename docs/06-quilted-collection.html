Showcase 06 · collection drops

One Walrus Quilt, 10 000 tokens, zero per-token pins.

OpenSea famously improved NFT metadata reliability by 99.2% when they switched to Pinata's infrastructure — i.e. the centralization tax for collection drops is enormous because each token needs its own pin. This contract eliminates that tax: pack the whole drop into one Walrus Quilt with filenames as identifiers, store one quiltId on-chain, and let tokenURI(id) deterministically synthesize the per-token aggregator URL.

01 · the pain it answers

"Decentralization theater" — one pin per token under one Pinata account.

"Your ownership is permanent and verifiable, but your ability to actually access what you own depends on services that could disappear tomorrow … OpenSea actually improved their NFT metadata reliability by 99.2% when they switched to Pinata's infrastructure." ipfs-pain.md §8 — decentralization theater

For a 10 000-token drop the operational shape on IPFS is brutal: 10 000 (image, metadata) pairs all pinned individually, all listed in a single pinning account, all silently dropable when that account lapses. The Walrus Quilt primitive answers exactly this: one blob holds many files, each addressable by its filename. The contract stores one id; the per-token URL is synthesized on-chain.

03 · read along

Constructor invariants, string concat tokenURI, owner-only migration knob

step 1 · storage layout

quiltId, aggregator, maxSupply — set once by constructor

Solidity doesn't allow immutable on dynamic-size types, so the quilt id (base64url string) and aggregator base URL live in normal storage with a constructor-only write contract. maxSupply is a real immutable on a fixed-size uint256.

The quiltId is stored already base64url-encoded — Move and Sui produce blob ids in that form natively, which sidesteps doing base64url encoding inside Solidity. The on-chain string is exactly the path segment the aggregator URL needs.

3 storage fields no per-token state OpenZeppelin ERC721

contract QuiltedCollection is ERC721, Ownable {
    using Strings for uint256;

    string  public quiltId;     // base64url, set once
    string  public aggregator;  // e.g. https://aggregator.../
    uint256 public immutable maxSupply;
    uint256 private _nextTokenId = 1;

    constructor(
        string memory name_,
        string memory symbol_,
        string memory quiltId_,
        string memory aggregator_,
        uint256 maxSupply_,
        address initialOwner
    ) ERC721(name_, symbol_) Ownable(initialOwner) {
        require(bytes(quiltId_).length    > 0, "empty quiltId");
        require(bytes(aggregator_).length > 0, "empty aggregator");
        require(maxSupply_              > 0, "zero maxSupply");
        quiltId    = quiltId_;
        aggregator = aggregator_;
        maxSupply  = maxSupply_;
    }
}

step 2 · tokenURI

string.concat — deterministic per-token URL, no state read

The whole magic of the showcase: tokenURI(id) assembles aggregator + "/v1/blobs/by-quilt-id/" + quiltId + "/" + id + ".json". That URL hits the Walrus aggregator's quilt patch endpoint, which serves the file stored inside the quilt under identifier "<id>.json".

The convention: when you pack the drop, name each file 1.json, 2.json, …, N.json for the metadata and 1.png, 2.png, …, N.png for the matching images. The metadata's image field references the matching .png inside the same quilt. _requireOwned(tokenId) enforces ERC-721's "token must exist" rule.

~10 LOC pure-string assembly deterministic
showcases/contracts/src/QuiltedCollection.sol (excerpt) view source ↗ 
function tokenURI(uint256 tokenId)
    public
    view
    override
    returns (string memory)
{
    _requireOwned(tokenId);   // ERC-721 existence check
    return string.concat(
        aggregator,
        "/v1/blobs/by-quilt-id/",
        quiltId,
        "/",
        tokenId.toString(),
        ".json"
    );
}

// resulting URL, for id = 42:
//   https://aggregator.walrus-testnet.walrus.space
//     /v1/blobs/by-quilt-id/<quiltId>/42.json
//
// Walrus aggregator serves the file whose identifier inside
// the quilt is "42.json" — no per-token pin, no per-token state.

step 3 · the migration knob

setAggregator(newAggregator) — owner-only, content stays content-addressed

The quilt itself is content-addressed; only the resolver URL prefix needs to be mutable in case the original aggregator host sunsets or the project migrates testnet → mainnet. setAggregator emits AggregatorUpdated(oldAggregator, newAggregator) for a clean audit trail.

What this is not: it isn't a pointer-to-content update. The quiltId never changes after deployment — the bytes a holder owns are exactly the bytes they minted. Only the host that serves them can be retargeted.

owner-only ~6 LOC content stays addressable
showcases/contracts/src/QuiltedCollection.sol (excerpt) view source ↗ 
event AggregatorUpdated(string oldAggregator, string newAggregator);

function setAggregator(string calldata newAggregator)
    external
    onlyOwner
{
    require(bytes(newAggregator).length > 0,
        "QuiltedCollection: empty aggregator");
    emit AggregatorUpdated(aggregator, newAggregator);
    aggregator = newAggregator;
}

// Migration story:
//   testnet → mainnet aggregator host
//   primary host sunsets, switch to a self-hosted mirror
// In both cases the quiltId stays the same, the bytes stay the same.
// Only the URL prefix changes — the same migration shape every Walrus
// showcase on this site uses.

05 · what to try next

Exercises

Compare gas cost vs a per-token _setTokenURI drop

Deploy the regular EvmWal NFT and mint 100 tokens; deploy this contract and mint 100 tokens. The quilt-backed version doesn't write per-token URI strings, so the per-mint gas savings show up cleanly. Multiply by your target supply.

GET a quilt patch URL by hand

curl https://aggregator.walrus-testnet.walrus.space/v1/blobs/by-quilt-id/<quiltId>/1.json. You should get the same JSON you packed into drop/1.json — no EVM, no contract, just a content-addressed fetch from a quilt.

Add a randomized minting order

Most real drops don't want sequential IDs revealed in order. Wrap _nextTokenId with a Fisher-Yates-style draw from a fixed pool — same quilt, same URLs, but unpredictable per-mint reveal. Watch out for the standard commit-reveal pitfalls.

Simulate the aggregator migration

Call setAggregator with a different host and confirm tokenURI(1) changes prefix while everything else stays identical. That's the "future-proof" half of the showcase made tangible.

Inspect the Walrus Quilt docs

docs.wal.app/walrus-client/quilts covers quilt grammar, size limits, and per-patch retrieval. Useful before authoring your own bulk packer.
