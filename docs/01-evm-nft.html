Showcase 01 · the reference dApp

A vanilla ERC-721 whose tokenURI lives on Walrus.

The EvmWal NFT is the canonical template the other five showcases riff on. The Solidity is an OpenZeppelin ERC721 + ERC721URIStorage with no Walrus knowledge — it just holds a string. The browser drops bytes on Walrus and writes the resulting aggregator URL into tokenURI. ~200 LOC end-to-end including a Next.js gallery and two write paths (operator-pays SDK or zero-infra public publisher).

Sources: EvmWalNFT.sol · handler.ts · useMint.ts Status: shipped at the repo root
01 · the pain it answers

"Don't trust free decentralized storage to outlive your project."

"Bankless's Metaversal newsletter now routinely frames IPFS as 'data must be pinned through services like ClubNFT or Pinata to be maintained over time,' and steers archival-grade projects toward Arweave or Walrus. NFT.Storage Classic stopped accepting uploads at 5 PM EST on June 30, 2024, affecting a stated 165,000 users and 660 TB of data." ipfs-pain.md §4 — NFT.Storage collapse fallout

The 2024 trust shock landed on NFT projects hardest: free pinning evaporated, paid SDKs churned, and every cancelled tokenURI became a broken card on every marketplace at once. The EvmWal NFT answers that with a contract whose EVM surface is unchanged from the IPFS era — _setTokenURI(tokenId, string) — pointing at a Walrus blob whose lifetime is bookkept on Sui as a first-class object rather than as opaque pin state inside a vendor.

03 · read along

Three files, ~120 lines, one pattern

The repo has more than three files, but these are the three that prove the thesis. Read them in order — contract, server handler, browser hook — and the migration story stops being hand-waving.

step 1 · the contract

EvmWalNFT.sol — the unchanged IPFS-era shape

Nothing in this contract knows about Walrus. _setTokenURI is the OpenZeppelin extension that an IPFS-backed NFT would already use; the string we hand it just happens to be an aggregator URL rather than ipfs://<cid>.

This is the leverage: every existing IPFS-backed NFT in production fits this same surface. Migration is changing what your frontend stores in this string, nothing else.

~55 LOC total 100% line coverage OpenZeppelin v5
contracts/src/EvmWalNFT.sol view source ↗ 
contract EvmWalNFT is ERC721, ERC721URIStorage, Ownable {
    uint256 private _nextTokenId = 1;

    event Minted(uint256 indexed tokenId,
                 address indexed minter,
                 string tokenURI_);

    function mint(string memory tokenURI_)
        external returns (uint256 tokenId)
    {
        return _mintWithURI(msg.sender, tokenURI_);
    }

    function _mintWithURI(address to, string memory uri)
        internal returns (uint256 tokenId)
    {
        tokenId = _nextTokenId++;
        _safeMint(to, tokenId);
        _setTokenURI(tokenId, uri);  // ← the whole Walrus thing
        emit Minted(tokenId, msg.sender, uri);
    }
}

step 2 · the upload route

handler.ts — one writeBlob, dependency-injected for tests

The Next.js API route at /api/walrus/upload is split into two files: route.ts wires the real WalrusClient from env vars, and this handler does the actual work against an injected deps.

Why split it: the tests pass a fake with walrusClient.walrus.writeBlob stubbed and assert behaviour without hitting the live SDK. epochs: 5 ≈ 14 days of testnet retention — same number the publisher path uses, so both modes produce identical blob lifetimes.

~50 LOC local-only · unauthenticated
web/app/api/walrus/upload/handler.ts view source ↗ 
export const MAX_UPLOAD_BYTES = 15 * 1024 * 1024;
export const STORAGE_EPOCHS = 5;

export async function handleWalrusUpload(
  bytes: Uint8Array,
  deps: WalrusServer,
): Promise<WalrusUploadResult> {
  if (bytes.length === 0) throw new EmptyUploadError();
  if (bytes.length > MAX_UPLOAD_BYTES) {
    throw new UploadTooLargeError(MAX_UPLOAD_BYTES, bytes.length);
  }

  const { blobId, blobObject } = await
    deps.walrusClient.walrus.writeBlob({
      blob: bytes,
      deletable: false,
      epochs: STORAGE_EPOCHS,
      signer: deps.keypair,
    });

  return { blobId, suiObjectId: blobObject.id };
}

step 3 · the browser orchestration

useMint.ts — image, metadata, mint, as one state machine

The mint UI is a five-state machine: uploading-image → uploading-metadata → awaiting-signature → awaiting-confirmation → success. The state machine is identical between backend and publisher modes — only the uploadBlob dispatcher (module-level constant) is different.

The key trick: aggregatorUrl(blobIdMetadata) is what ends up in the contract. The image blob id is referenced from inside the metadata JSON, never directly on-chain. Two blobs, one EVM string, one ERC-721 standard.

~165 LOC wagmi v2 · viem module-level dispatcher
web/hooks/useMint.ts (excerpt) view source ↗ 
// dispatch picked once at module load from NEXT_PUBLIC_WALRUS_UPLOAD_MODE
const uploadBlob =
  process.env.NEXT_PUBLIC_WALRUS_UPLOAD_MODE === 'publisher'
    ? uploadWalrusBlob              // PUT to public publisher
    : uploadWalrusBlobViaBackend;   // POST to local route

async function submit(input: MintInput) {
  setStatus('uploading-image');
  const imageRes = await uploadBlob(bytes, input.file.type);

  setStatus('uploading-metadata');
  const meta = buildMetadata({ blobIdImage: imageRes.blobId, ... });
  const metaRes = await uploadBlob(
    JSON.stringify(meta), 'application/json');

  setStatus('awaiting-signature');
  const tokenURI = aggregatorUrl(metaRes.blobId);
  const hash = await writeContractAsync({
    address: CONTRACT_ADDRESS, abi: CONTRACT_ABI,
    functionName: 'mint', args: [tokenURI],
  });
  setTxHash(hash);
  setStatus('awaiting-confirmation');
}
04 · run it locally

Two terminals, three pnpm commands

Zero-Sui-setup path

The fastest way to see a mint complete end-to-end. Uses the public publisher — no Sui keypair, no faucet, no API route. Just clone, install, set one env var, and run.

terminal · publisher mode
# clone + install
git clone git@github.com:MystenLabs/evm-nft-wal.git
cd evm-nft-wal
pnpm install

# enable publisher mode in web/.env.local
cp web/.env.local.example web/.env.local
echo 'NEXT_PUBLIC_WALRUS_UPLOAD_MODE=publisher' >> web/.env.local

# terminal A — local chain
pnpm dev:chain

# terminal B — deploy + run web
pnpm deploy:local
pnpm extract-abi
pnpm dev:web

Operator (backend) path

Mirrors the production write shape: @mysten/walrus in a server-side route, operator-funded WAL, an on-Sui Blob object you own and can extend. Needs a Sui testnet keypair with both SUI gas and WAL storage.

terminal · backend mode
# 1. testnet SUI from https://faucet.sui.io/
# 2. testnet WAL from the Walrus docs faucet
# 3. drop SUI_PRIVATE_KEY into web/.env.local
#    (bech32, starts with suiprivkey...)

# same three pnpm commands
pnpm dev:chain
pnpm deploy:local && pnpm extract-abi
pnpm dev:web

# open http://localhost:3000
# connect MetaMask · switch to Anvil (chain 31337)
# go to the Mint tab and drop a PNG

Backend mode's API route is local-only and unauthenticated. It signs Sui transactions with your keypair and spends WAL on every successful request — no shared secret, no rate limit. Run it against a localhost-only dev server with a testnet keypair you control.

05 · what to try next

Exercises that turn this page into learning material

Flip between publisher and backend mode without redeploying

Toggle NEXT_PUBLIC_WALRUS_UPLOAD_MODE in web/.env.local and reload. The minted NFTs render identically — the contract sees no difference. That's the migration cost made tangible.

Open a minted token's blob on Walruscan

Click any card in the gallery. walruscan.com/testnet shows the blob, its on-Sui Blob object, and its remaining epoch retention. In backend mode the post-mint panel also deep-links the Blob object on Suiscan — that's the on-chain truth IPFS never had.

Inspect the metadata JSON the dApp built

tokenURI(id) returns the aggregator URL of the metadata blob. GET it with curl and you'll see the ERC-721 metadata schema with image pointing at a second aggregator URL — two blobs, one EVM string.

Read contracts/test/EvmWalNFT.t.sol

Seven Foundry tests, 100% line coverage on the contract. Notice how none of them mention Walrus — they assert ERC-721 behaviour over arbitrary strings, which is exactly what makes the migration story credible.

Try the handler tests with a fake WalrusServer

The dependency-injected handler in handler.ts exists specifically so the upload-route tests can stub writeBlob without hitting the live SDK. Look at the test suite under tests/ — it's a template for testing any backend that talks to @mysten/walrus.
