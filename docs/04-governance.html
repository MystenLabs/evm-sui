Showcase 04 · DAO proposals

Proposal bodies on Walrus, proposer-pays storage, on-chain tallies.

Snapshot famously runs governance for Aave and Uniswap on top of 4Everland-pinned IPFS — bundled hosted IPFS that absorbs the operational pain because public gateways can't be trusted. This contract keeps the same shape but stores just the 32-byte blobId, deadline, and vote tallies. The proposer uploads the body to Walrus using the public publisher, so the DAO contract is never on the WAL hook.

Sources: Governance.sol Status: implemented · showcase-only voting surface
01 · the pain it answers

Hosted IPFS sits under every DAO vote.

"4Everland is used by Snapshot governance for Aave and Uniswap DAO votes" — bundled into the same hosted-IPFS tier that absorbs the operational pain because public gateways can't be trusted. ipfs-pain.md §9 — hosted IPFS providers

The current arrangement works until it doesn't: a single hosted-IPFS provider sits beneath every Snapshot proposal anyone reads. If that provider rate-limits, expires a pin, or has a vendor outage on vote day, the proposal body becomes unverifiable bytes — even though the on-chain tally keeps ticking. This contract decouples the body from the vendor: proposer publishes anonymously to Walrus's public publisher, contract stores a content-addressed blob id, anyone reads via the aggregator.

03 · read along

Proposal struct, propose, vote, tally — with one big warning.

step 1 · the storage shape

Proposal struct — proposer, blobId, deadline, two tallies

Five fields per proposal. The body is referenced by its 32-byte content address — no IPFS CID string, no IPNS pointer. uint128 tallies leave room for huge balances without burning a full slot; uint64 deadline is unix seconds because Solidity's block.timestamp is.

What's not here: snapshot block, voting strategy id, any per-voter data beyond a yes/no flag. This contract deliberately keeps the on-chain surface tiny so the Walrus-integration shape is legible — see the warning in step 3 for why production deployments need more.

~100 LOC sol no off-chain dependency
showcases/contracts/src/Governance.sol (excerpt) view source ↗ 
contract Governance {
    struct Proposal {
        address proposer;
        bytes32 blobId;    // Walrus blob holding the body
        uint64  deadline;  // voting closes when block.timestamp >= deadline
        uint128 yes;       // sum of voter weights for YES
        uint128 no;        // sum of voter weights for NO
    }

    IERC20 public immutable voteToken;
    uint256 public lastProposalId;

    mapping(uint256 id => Proposal) public proposals;
    mapping(uint256 id => mapping(address voter => bool)) public hasVoted;

    event Proposed(uint256 indexed id, address indexed proposer,
                   bytes32 blobId, uint64 deadline);
    event Voted(uint256 indexed id, address indexed voter,
                bool support, uint256 weight);
}

step 2 · propose

propose(blobId, deadline) — pointer-only, no body on-chain

The proposer has already uploaded the body to Walrus before calling propose. The contract validates the deadline is in the future and the blobId isn't zero, then stores the proposal and emits Proposed with the blobId in the log so indexers and frontends can discover proposals without scanning state.

The migration story: from a Snapshot operator's perspective, this is exactly their existing proposal-creation flow — sign a typed message that includes a content reference and a deadline — except the content reference is a Walrus blob id instead of an IPFS CID.

~12 LOC body deadline check zero-blobId guard
showcases/contracts/src/Governance.sol (excerpt) view source ↗ 
function propose(bytes32 blobId, uint64 deadline)
    external
    returns (uint256 id)
{
    require(deadline > block.timestamp,
        "Governance: deadline in past");
    require(blobId != bytes32(0),
        "Governance: zero blobId");

    id = ++lastProposalId;
    proposals[id] = Proposal({
        proposer: msg.sender,
        blobId:   blobId,
        deadline: deadline,
        yes:      0,
        no:       0
    });
    emit Proposed(id, msg.sender, blobId, deadline);
}

step 3 · vote — and the flash-loan caveat

vote(id, support) — weight = balanceOf, with a giant production warning

The voting surface is intentionally minimal: a yes/no vote weighted by the caller's live voteToken.balanceOf(msg.sender). That keeps the file readable — but it also makes the contract flash-loan-attackable against any ERC-20 with a flash-mint or flash-borrow hook (most modern DEX-paired tokens).

Don't deploy this as written. Pair the contract with an OpenZeppelin ERC20Votes token, snapshot the proposal's start block, and read voting weight via IVotes(voteToken).getPastVotes(msg.sender, startBlock). That's the only correct way to use this pattern against an asset that can be borrowed in a single transaction.

flash-loan attackable as written fix: ERC20Votes + getPastVotes
showcases/contracts/src/Governance.sol (excerpt) view source ↗ 
function vote(uint256 id, bool support) external {
    Proposal storage p = proposals[id];
    require(p.deadline != 0,           "unknown proposal");
    require(block.timestamp < p.deadline, "voting closed");
    require(!hasVoted[id][msg.sender],     "already voted");

    // ⚠ FLASH-LOAN ATTACKABLE — balanceOf reads the live balance.
    // Production: snapshot the proposal start block, read via
    //   IVotes(voteToken).getPastVotes(msg.sender, startBlock)
    // and pair with an ERC20Votes token.
    uint256 weight = voteToken.balanceOf(msg.sender);
    require(weight > 0, "zero weight");
    require(weight <= type(uint128).max, "weight overflow");

    hasVoted[id][msg.sender] = true;
    if (support) p.yes += uint128(weight);
    else         p.no  += uint128(weight);
    emit Voted(id, msg.sender, support, weight);
}

step 4 · tally

tally(id) — convenience view returning (yes, no, passed, closed)

A frontend wants four pieces of information at once: the two raw counts, whether voting has closed, and whether the proposal passed. tally packages all four into a single call, with a deliberate revert on unknown ids so probing for existence requires comparing against lastProposalId first — which keeps logs honest.

What passed means here is intentionally simple: more YES weight than NO weight when the deadline elapses. Real DAOs almost always want a quorum and a supermajority on top — both clean follow-on exercises against this contract.

~10 LOC no quorum (yet)
showcases/contracts/src/Governance.sol (excerpt) view source ↗ 
function tally(uint256 id)
    external
    view
    returns (uint128 yes, uint128 no, bool passed, bool closed)
{
    Proposal storage p = proposals[id];
    require(p.deadline != 0, "unknown proposal");
    yes    = p.yes;
    no     = p.no;
    closed = block.timestamp >= p.deadline;
    passed = closed && yes > no;
}

05 · what to try next

Exercises

Replace balanceOf with getPastVotes

Add OpenZeppelin's ERC20Votes to the test setup, store startBlock on the Proposal struct, and switch vote() to read past balances. That alone makes the contract flash-loan-safe.

Add a quorum threshold

Pass a quorum parameter to the constructor (or per-proposal); change passed in tally() to require yes + no >= quorum. Cheap, useful, near-universal in real DAOs.

Build a viewer that fetches and renders the proposal body

const blobId = proposals(id).blobId then fetch(`${AGGREGATOR}/v1/blobs/${base64url(blobId)}`) → render markdown. Compare latency against a Snapshot proposal hosted on 4Everland-IPFS.

Index Proposed and Voted with viem

Both events have indexed topics for cheap filtering. A 20-line indexer is enough to power a full DAO dashboard — useful as a comparison point against running a Subgraph.

Read the NatSpec at the top of the file in full

The flash-loan warning, the fix recipe, and the scope disclaimer all live in the contract header. That's the part you'd want to read aloud during a security review — it tells you what the showcase is and isn't.
