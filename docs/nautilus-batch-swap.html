nautilus · N3

Sealed batch swap: one clearing price, no sandwich.

CoW Protocol proved that batch auctions eliminate sandwich attacks: all orders in a batch execute at the same uniform price. This example runs the same idea inside a Nautilus TEE — orders are invisible until the crossing match is complete. The enclave sorts buys descending and sells ascending, finds the crossing point, computes a midpoint clearing price, and signs the result. Sui verifies, Walrus stores the proof, and a Solidity stub shows EVM settlement.

Stack: Rust · Move · Solidity stub Source: nautilus/ Local: Sui localnet · Anvil · enclave on localhost:3000
01 · the problem

Every AMM swap is a sandwich waiting to happen.

On a constant-product AMM the trade's price impact is visible in the mempool before execution. MEV bots insert a buy before your swap (pushing the price up) and a sell after (pocketing the difference). The user gets a worse price; the bot profits from the slippage.

Batch auctions solve this by collecting orders during a discrete window, then executing them all at a single uniform clearing price. There is no ordering within the batch, so there is no front or back to sandwich. Running the batch auction inside a TEE adds an extra guarantee: the order book is invisible to everyone — including the operator — until the clearing price is computed and signed.

Sandwich attack surface. Any transparent mempool where trade size and direction are visible before execution is vulnerable. A sealed batch auction inside a TEE removes the information asymmetry entirely — no one sees the order book until the match is final.
02 · the shape

Crossing-order match at a uniform clearing price.

Participants submit buy and sell Intent structs (each with a side, limit_price, and amount) to the enclave over TLS during an auction window. The enclave runs a crossing-order match: buys sorted descending by limit price, sells sorted ascending. If the best buy >= the best sell, a crossing exists. The clearing price is the midpoint of the best buy and best sell. All eligible orders fill at that price.

The result — a BatchSwapPayload with the clearing price, total filled volumes, and participant count — is BCS-serialized, signed, and posted to Sui. Per-participant fills are stored in the Walrus attestation blob for independent verification.

03 · move contract

BatchSwapPayload and settle_batch.

The batch_swap::batch_swap module defines a BatchSwapPayload carrying the aggregate batch result. The settle_batch function reconstructs the payload, verifies the enclave signature, and creates a shared SettledBatch object.

Note: the Move contract stores only the aggregate — clearing price, total buy/sell filled, participant count. Individual per-participant fills live in the Walrus blob referenced by walrus_proof_blob_id, keeping on-chain storage minimal.

04 · enclave logic

Crossing match: sort, cross, midpoint.

The crossing_match function implements the core algorithm:

  • Separate intents into buys (side == 0) and sells (side == 1).
  • Sort buys descending by limit_price, sells ascending.
  • Check crossing: if best_buy < best_sell, no match is possible.
  • Clearing price = (best_buy + best_sell) / 2.
  • Fill all buys with limit_price >= clearing_price and all sells with limit_price <= clearing_price.

The result is BCS-encoded and signed. The Walrus attestation blob includes per-participant fill details so anyone can verify their allocation independently. 

// enclave-server/src/batch_swap/mod.rs
fn crossing_match(
    intents: &[Intent]
) -> Option<(u64, ..)> {
    let mut buys = intents.iter()
        .filter(|i| i.side == SIDE_BUY)
        .collect();
    let mut sells = intents.iter()
        .filter(|i| i.side == SIDE_SELL)
        .collect();

    // Sort buys desc, sells asc
    buys.sort_by(|a, b| b.limit_price.cmp(..));
    sells.sort_by(|a, b| a.limit_price.cmp(..));

    // Clearing price = midpoint
    let clearing =
        (buys[0].limit_price
       + sells[0].limit_price) / 2;

}
05 · solidity stub

BatchSwapSettlement: the EVM side.

// contracts/src/BatchSwapSettlement.sol — showcase stub
contract BatchSwapSettlement {
    struct Batch {
        bytes32 batchId;
        bytes32 tokenPair;
        uint256 clearingPrice;
        uint256 totalBuyFilled;
        uint256 totalSellFilled;
        uint16  numParticipants;
        bytes32 proofBlobId;   // Walrus attestation
        uint64  settledAt;
    }

    function settleBatch(
        bytes32 batchId, bytes32 tokenPair,
        uint256 clearingPrice, ...
    ) external;

    event BatchSettled(bytes32 indexed batchId,
        bytes32 tokenPair, uint256 clearingPrice,
        uint16 numParticipants);
}

The EVM contract records only the aggregate batch result. Per-participant fills are stored in the Walrus attestation blob referenced by proofBlobId. Production would verify an EIP-712 signature via ecrecover.

06 · run locally

Build, start, verify.

01

Build the Move packages

cd nautilus/move/enclave && sui move build
cd ../batch-swap && sui move build
02

Build and test the Rust enclave server

cd nautilus/enclave-server
cargo test --no-default-features --features batch-swap
cargo run --no-default-features --features batch-swap
# → Enclave pubkey: ef56...
# → listening on 0.0.0.0:3000

Only one feature can be active at a time. Use --no-default-features to disable the default rfq feature.

03

Build and test the Solidity stubs

cd nautilus/contracts
forge test -vv --match-contract BatchSwapSettlement
04

Submit a test batch

curl -X POST http://localhost:3000/process_data \
  -H "Content-Type: application/json" \
  -d '{"payload":{"batch_id":[187],"token_pair":[16,32],"intents":[{"participant":[1],"side":0,"limit_price":110,"amount":50},{"participant":[2],"side":0,"limit_price":105,"amount":30},{"participant":[3],"side":1,"limit_price":100,"amount":40},{"participant":[4],"side":1,"limit_price":108,"amount":20}]}}'
# → { "response": { ..., "data": { "clearing_price": 105, "total_matched_amount": 80, ... } }, "signature": "..." }

Best buy = 110, best sell = 100, clearing price = (110+100)/2 = 105. Both buys fill (110 >= 105, 105 >= 105); sell at 100 fills (100 <= 105); sell at 108 does not (108 > 105).

07 · verify

What "good" looks like.

  • cargo test --no-default-features --features batch-swap passes all tests including crossing-match logic and BCS round-trip.
  • The /process_data endpoint returns a clearing price of 105 for the test fixture (best buy 110, best sell 100).
  • No-crossing case (best buy < best sell) returns a clean error, not a panic.
  • forge test -vv passes the BatchSwapSettlement tests — settle once succeeds, duplicate reverts.
  • The BCS hex from the Rust test matches what the Move BatchSwapPayload would produce for the same inputs.
