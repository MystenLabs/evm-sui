Showcase 05 · token lists & dApp manifests

Resolve tokens.uniswap.eth in one eth_call + one GET.

The Uniswap token list is the canonical example of a small, high-read, cache-friendly JSON blob whose IPFS-pinned hosting flaked enough to trigger interface issue #7753 (rewrite every Cloudflare gateway reference). This showcase reuses WalrusResolver from showcase 03 plus a strict JSON schema: one eth_call resolves the pointer, one aggregator GET returns the bytes, JSON.parse closes it out. ~80 LOC of TypeScript, no SDK on the read path.

Sources: manifest.ts · example-usage.ts Status: implemented · reuses showcase 03
01 · the pain it answers

Public gateways flake; token lists are exactly where that hurts.

"ipfs.io … 85–90% of the time, it resolves … sometimes takes 5 min" — and Uniswap's interface release notes broke because public gateways flake. Token lists and dApp manifests are exactly the small, high-read, cache-friendly artifact that public-gateway flakiness ruins. ipfs-pain.md §2 — gateways

Wallets and DEX frontends fetch token lists on every cold start; if the lookup is an IPNS resolution plus an IPFS gateway GET, both stages can stall on any given visit. Replacing the pointer with an EVM read is the entire leverage: the resolver is deterministic instead of a DHT lottery, the aggregator is a normal HTTPS cache target, and the bytes are content-addressed by the same blob id the EVM call returned.

Two network round-trips. The first is a sub-200 ms RPC call. The second is a normal HTTPS GET your CDN can already cache.

Trust boundary

The aggregator serves the bytes that correspond to the requested blob id — that's the same trust assumption as fetching by CID from an IPFS gateway. The leverage over IPFS lives in the pointer: one deterministic EVM read instead of an IPNS lottery.

For true trustless retrieval, recompute the blob id from the bytes using @mysten/walrus's Reed-Solomon encoder client-side and assert equality. The showcase keeps that out of the read path on purpose — read-path simplicity is the migration story.

03 · read along

Three short pieces, all in TypeScript

step 1 · the ABI fragment

WALRUS_RESOLVER_ABI — just one function, viem-typed

Showcase 03's contract has more methods, but this client only needs walrusBlob(bytes32). Declaring the ABI inline is the right call for a single-purpose client — no codegen step, no full-contract drift, and viem infers the return tuple types from the literal.

const WALRUS_RESOLVER_ABI = [
  {
    type: "function",
    name: "walrusBlob",
    stateMutability: "view",
    inputs:  [{ name: "node", type: "bytes32" }],
    outputs: [
      { name: "blobId",      type: "bytes32" },
      { name: "suiObjectId", type: "bytes32" },
      { name: "contentType", type: "bytes8"  },
    ],
  },
] as const;

step 2 · resolveManifest

One generic function, four failure modes

resolveManifest<T> is the whole product surface. It does four things in sequence: namehash the ENS string, read the pointer, validate the on-chain contentType shortcode, then GET and parse. Each step throws a specific error so callers can route on it.

The contentType check is the interesting part: showcase 03 stores an 8-char ASCII shortcode in bytes8. This client refuses to parse anything that doesn't start with "app/json". Long MIMEs like application/json don't fit and are intentionally out of grammar.

~40 LOC generic over T four failure modes
showcases/05-verifiable-manifest/manifest.ts (excerpt) view source ↗ 
export async function resolveManifest<T = unknown>(
  ensName: string,
  opts: ResolveOpts,
): Promise<ResolvedManifest<T>> {
  const evm = createPublicClient({
    chain: mainnet, transport: http(opts.rpcUrl),
  });

  const node = namehash(ensName);
  const [blobId, , contentTypeRaw] = await evm.readContract({
    address: opts.resolverAddress,
    abi: WALRUS_RESOLVER_ABI,
    functionName: "walrusBlob",
    args: [node],
  });

  if (blobId === ZERO_BYTES32)
    throw new Error(`pointer unset for ${ensName}`);

  const contentType = asciiTrim(contentTypeRaw);
  if (!contentType.startsWith("app/json"))
    throw new Error(`unexpected contentType "${contentType}"`);

  const url = `${opts.aggregator}/v1/blobs/${bytes32ToBase64Url(blobId)}`;
  const res = await fetch(url);
  if (!res.ok)
    throw new Error(`aggregator HTTP ${res.status}`);

  return {
    manifest:    JSON.parse(await res.text()) as T,
    blobId,
    contentType,
  };
}

step 3 · the base64url helper

bytes32ToBase64Url — bridging EVM and Walrus URL conventions

EVM hands you bytes32; Walrus aggregator URLs use canonical base64url (no padding). The helper does the conversion in vanilla browser-runtime APIs — hexToBytes + btoa with the +/= replacements. Same convention Move and Sui produce natively for blob ids, so the round-trip preserves the content address.

~6 LOC browser-runtime only
showcases/05-verifiable-manifest/manifest.ts (excerpt) view source ↗ 
function bytes32ToBase64Url(hex: `0x${string}`): string {
  const bytes = hexToBytes(hex, { size: 32 });
  let bin = "";
  for (const b of bytes) bin += String.fromCharCode(b);
  return btoa(bin)
    .replace(/\+/g, "-")
    .replace(/\//g, "_")
    .replace(/=+$/, "");
}

// asciiTrim() — read a `bytes8` shortcode off the chain, drop the
// trailing zeros so 'app/json\0\0\0\0' becomes 'app/json'.
function asciiTrim(b: `0x${string}`): string {
  const bytes = hexToBytes(b, { size: 8 });
  const end = bytes.indexOf(0);
  return new TextDecoder("ascii").decode(
    end === -1 ? bytes : bytes.subarray(0, end)
  );
}

05 · what to try next

Exercises

Type a different manifest schema and pass it as the generic T

resolveManifest<MyShape>(name, opts) gives you a fully typed response — no codegen, no runtime cost. Useful for dApp config files, RSS feeds, or any JSON artifact whose schema is known.

Add a Reed-Solomon-backed integrity check on top

Import @mysten/walrus, encode the returned bytes with the same parameters Walrus used, recompute the blob id, and assert equality before JSON.parse. That closes the aggregator trust gap mentioned in section 02.

Cache the resolved blob id by ENS name

The pointer only changes when the ENS owner explicitly updates it; for read-heavy paths a short-TTL in-memory cache keyed by namehash(ensName) is safe and cuts the RPC traffic in half. Wallets and indexers should default this on.

Subscribe to WalrusBlobChanged for cache invalidation

From showcase 03's contract. Filter by indexed node equal to your namehash; on each event, drop the cache entry. That's the full picture of "deterministic mutable pointer" with no IPNS in sight.

Build a viewer with a 10-token preview + diff

Resolve the same ENS name from two different aggregators (testnet + a self-hosted mirror) and compare. They should agree exactly — same blob id, same bytes. If they don't, one of the aggregators is misbehaving and you've discovered the value of client-side integrity checks.
