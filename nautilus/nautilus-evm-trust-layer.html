The Reframe

The "must leave EVM" claim is true if Nautilus is a replacement. It's not true if Nautilus is a complementary verification layer that EVM dapps consume.

The expensive part of Nautilus — verifying the AWS certificate chain and PCR attestation — costs ~70M gas on Solidity (Marlin had to optimize like hell to get it that low). On Sui/Move, it's cheap enough to do at registration. So the architecture becomes:

  • Sui handles attestation verification (the hard crypto) + PCR governance (key rotation as Move transactions)
  • The enclave signs results that both chains can consume — Ed25519 for Sui, secp256k1 for EVM
  • Solidity trusts the enclave's pubkey because it's anchored on Sui with verified PCRs

That's genuinely something you can't get from Marlin-on-EVM-alone or Phala-alone — the Sui side acts as a lightweight, cheap, first-party verification root that the EVM side doesn't have to replicate.

1. The Dual-Chain Architecture

The cost asymmetry that makes this work: AWS cert chain verification costs ~70M gas on Solidity (Marlin's optimized Nitro verifier). On Sui/Move, it's cheap enough to do at registration. So Sui becomes the verification root that EVM doesn't have to replicate. The EVM side just runs ecrecover() against a known pubkey — negligible gas.

2. The Full Stack — What No Single EVM Solution Offers

Why this combination is unique: Chainlink Functions can't do Seal (no threshold encryption). Phala doesn't have Seal (different encryption model). Oasis Sapphire does full confidentiality but can't do cross-chain settlement on Ethereum L1. This stack lets you encrypt (Seal), compute privately (Nautilus), verify cheaply (Sui), and settle where the money is (EVM).

The "not fully confidential" weakness from the previous analysis becomes addressable here. Seal encrypts data at rest, Nautilus decrypts only inside the TEE, and only the cleartext result crosses to EVM for settlement. The matching/computation is private; the settlement is public. For the "private matching, public settlement" pattern — dark pools, sealed-bid auctions, confidential DeFi — this is exactly the right split.

4. Honest Weaknesses

Bridge Trust

You're adding Wormhole/Axelar/LayerZero bridge assumptions on top of TEE trust. Every bridge is an attack surface. KelpDAO lost $293M to a DVN bridge failure. The bridge between Sui and EVM is a real, not theoretical, risk vector.

Dual-Sign Is Untested

Having the enclave sign with Ed25519 + BCS for Sui and secp256k1 + ABI for EVM is architecturally clean. But nobody has shipped this in production yet. BCS field ordering bugs are hard enough on one chain — two serialization formats doubles the surface.

Seal Is Early

Threshold key management is live. The "Seal + Nautilus combined pattern" is documented in Sui docs as a design pattern, not as a battle-tested production flow. The key server network is young. Real-world stress testing under adversarial conditions is limited.

Why Not Just Marlin Oyster from EVM?

Oyster already does PCR verification on Solidity (~70M gas, optimized). The Sui layer only adds value when you need Seal (encryption), or when PCR governance / key rotation benefits from Move's safety model. For pure attested compute, Oyster-from-EVM is simpler.

5. When This Makes Sense (and When It Doesn't)

Use Nautilus + EVM When

  • You need encrypted state (Seal) + attested compute (Nautilus) + DeFi settlement (EVM)
  • Confidential matching engines: matching is private (Sui side), settlement is public (EVM side)
  • PCR governance / key rotation benefits from Move's safety model
  • App-specific oracles feeding verified data to both Sui and EVM dapps
  • You're already building on Sui and want to reach EVM liquidity

Don't Bother When

  • Pure EVM oracle needs: just use Marlin Oyster from EVM directly
  • No encryption needed: TEE alone works fine from EVM via Phala/Oyster
  • Bridge risk is unacceptable for your threat model
  • You don't need the Sui state management layer
  • Fully confidential EVM is the goal — Oasis Sapphire is purpose-built

The Decision Heuristic

Ask yourself: "Does my problem need encryption at rest (Seal) or Move-safety for governance?"

  • Yes to either → Nautilus + Sui + EVM is worth the complexity. The Sui layer earns its keep.
  • No to both → Use Marlin Oyster or Phala directly from EVM. Adding Sui is unnecessary architecture.
Nautilus isn't an EVM replacement — it's the missing trust layer that EVM dapps can plug into when they need attested compute + encrypted state that Solidity alone can't provide.
