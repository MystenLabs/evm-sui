1. What Nautilus Actually Is

Nautilus = a framework for running Rust code inside AWS Nitro Enclaves, where a Move contract on Sui cryptographically verifies every output the enclave produces.

It is not a blockchain, not a rollup, not a sidechain. It's a pattern: trusted hardware signs data, an on-chain contract checks the signature. The novelty is how tight the coupling is between the enclave and the chain.

EVM mental model: Think of it like Chainlink Functions, but instead of a committee of DON nodes voting on what's true, you have a single machine running inside tamper-proof hardware that mathematically proves which binary it's executing. The Move contract doesn't trust the machine — it trusts the math.

The PCR System (Platform Configuration Registers)

PCRs are SHA-384 hashes baked into the enclave image. They are Nautilus's equivalent of a contract's bytecode hash on Ethereum — but they cover the entire runtime, not just the application code.

PCR0 — OS & Boot

  • Hash of the enclave's operating system image
  • Immutable for a given AWS AMI
  • Like the EVM version your contract runs on

PCR1 — Application Code

  • Hash of your compiled Rust binary (EIF image)
  • Change one byte of code → PCR1 changes
  • Like keccak256(bytecode) for your off-chain logic

PCR2 — Runtime Config

  • Hash of run.sh, traffic forwarding rules
  • Proves the enclave's network setup wasn't tampered with
  • No EVM equivalent — new concept

The Guarantee

  • Anyone can rebuild from source and compare PCRs
  • If PCRs match → binary matches published source
  • Move contract stores expected PCRs; rejects mismatches

2. How It Uses Sui — The Two-Phase Pattern

The core insight: expensive verification happens once at registration; cheap signature checks happen on every call forever after.

Key insight for EVM devs: The BCS (Binary Canonical Serialization) encoding is critical. Rust struct field order must match Move struct field order exactly — BCS is positional, not name-keyed. Mismatch → signature verification silently fails. There's no ABI encoding/decoding hop like Solidity↔ethers.js — payloads are encoded the same way Sui encodes its own objects.
How BCS coupling works (Rust vs Move)

BCS serializes structs by field order. The Rust side and the Move side must declare fields in the exact same order:

// Rust (enclave side)
struct PriceResponse {
    coin_id: String,      // field 0
    vs: String,           // field 1
    price_micro: u64,     // field 2
}

// Move (chain side) — MUST match field order
struct PricePayload has copy, drop {
    coin_id: vector<u8>,  // field 0
    vs: vector<u8>,       // field 1
    price_micro: u64,     // field 2
}

No field names in the wire format — just bytes in order. Swap vs and price_micro on either side and signature verification fails silently.

3. Nautilus vs Nitro vs Oyster — "Is It Still Nautilus?"

This is the part that confuses everyone. Three names, three different things, all involved in the same system.

Running on Oyster = still Nautilus?

  • Yes. Same Rust code, same PCR values, same Move verification contracts
  • Oyster operators handle AWS provisioning, networking, billing
  • Your Move contract can't tell the difference — it just checks pubkey + PCRs

Could Nautilus run on non-Nitro TEEs?

  • Not today. Attestation format is Nitro-specific (NSM driver, AWS cert chain)
  • Mysten: "Support for additional TEE providers may be considered in the future."
  • Move verification contracts would need rewriting for SGX/TDX attestation formats
The trust root question: Whether you self-manage AWS or use Oyster, the ultimate trust anchor is AWS's Nitro root certificate. If Nitro has a zero-day, both paths break. This is the single most important threat model assumption. If multi-cloud TEE diversity is mandatory, look at Phala (SGX/TDX/SEV/H100 roadmap).

Where Nautilus Wins

Tightest TEE-to-Chain Coupling

BCS encoding — same serialization Sui uses for its own objects. No JSON→ABI-encode→verify hops. Move verifier deserializes payloads directly.

Reproducible Builds by Default

Nix-style template ships with the framework. PCRs are deterministic from source. Anyone can rebuild and verify.

Free Local Dev Mode

Full Rust + Move dev loop with zero AWS spend. No Phala worker fees, no SGX cloud time. Pay only in production.

First-Party Maintenance

Mysten Labs builds both the framework and the Sui runtime. Bug fixes and docs go through one team. No multi-vendor integration tax.

Where Nautilus Loses

Must Leave EVM

You need Move + Rust. If your users, liquidity, or partners are on Ethereum — Phala, Oyster-from-EVM, or Sapphire are better.

Selective, Not Full Confidentiality

Move contract on Sui stays public. Only enclave contents are private. For "everything encrypted by default" → Oasis Sapphire wins.

AWS Nitro Lock-in

Single TEE vendor today. Multi-cloud TEE diversity? Phala's SGX/TDX/SEV/H100 roadmap is broader.

~4-8 Week Ramp-Up

Move's resource semantics + Rust for enclaves + AWS Nitro ops. Real cost for Solidity/TS teams.

5. What "Mocking" Nautilus Locally Actually Means

When you "mock" Nautilus locally, you're replacing the hardware trust guarantees with deterministic test values while keeping the entire signing-to-verification pipeline functional.

The Local Dev Loop

# 1. Run Rust server locally (no AWS needed)
cargo run --features=price-oracle --bin nautilus-server

# 2. Hit the process endpoint
curl http://localhost:3000/process_data -d '{"coin_id":"sui"}'
# → returns {response: ..., signature: ...}

# 3. Submit signed payload to Move contract on Sui testnet
sui client call --package 0x... --module price_oracle \
  --function update_price --args ... $SIGNATURE

# 4. Move contract runs verify_signature()
#    Passes using all-zero PCRs + deterministic pubkey
# Total AWS cost: $0
What this validates: Your entire Rust→BCS→sign→Move verify pipeline works end-to-end. BCS field ordering bugs, payload format mismatches, signature encoding issues — all caught locally, for free. The only thing you can't test locally is the AWS attestation document.
When to graduate: Once the local loop works, deploy to real AWS Nitro (~$0.19/hr self-managed) or Marlin Oyster (pay-per-minute). Remove the deterministic seed, use real PCRs from your reproducible build, call register_enclave.sh with a real attestation document. The Move contract doesn't change — only the PCR values and pubkey do.

6. Decision Flow — Should I Use Nautilus?

TL;DR — The One-Paragraph Summary

Nautilus is Mysten Labs' framework for running Rust code inside AWS Nitro Enclaves with cryptographic proof on Sui. The enclave generates a keypair (private key never leaves hardware), signs every output with BCS encoding, and a Move contract verifies the signature against a registered pubkey + PCR hashes that commit to the exact binary running inside the enclave. AWS Nitro is the hardware; Marlin Oyster is an optional managed marketplace for that hardware; running on either is "still Nautilus" because the framework (Rust template + Move contracts + reproducible build pipeline) stays the same. "Mocking locally" means replacing the ephemeral key with a deterministic seed and PCRs with all-zeros, which gives you a full signing-to-verification dev loop for $0 — you just don't get real attestation guarantees until you deploy to actual Nitro hardware. Compared to EVM solutions: Chainlink Functions has no TEE, Phala has broader TEE diversity but looser chain coupling, Oasis Sapphire is fully confidential EVM (Nautilus is selective), and Flashbots SUAVE is Ethereum-MEV-specific. Nautilus's edge is the tightest TEE-to-chain integration via BCS, reproducible builds by default, and first-party Mysten maintenance — but the cost is leaving EVM and learning Move + Rust.
